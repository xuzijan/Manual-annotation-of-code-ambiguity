## **控制标签**

在进入 6 大歧义分类之前，标注者必须首先判断该 Prompt 是否真的存在“歧义”。请严格使用以下两个控制标签过滤无效数据：
• Not Ambiguous (无歧义 / 纯粹的模型能力不足)
定义： 用户指令清晰、意图唯一。如果 LLM 生成错误，是因为模型的逻辑推理能力不足或产生幻觉，而非指令本身允许多种解释。大白话：用户真的对的。
• Other (其他 / 无法归类)
定义： 确实存在导致多种代码实现路径的歧义，但无法被纳入下方定义的 6 个核心类别中。比如本身提示词就是错的。llm由于幻觉在错误的基础上继续延申错误。
备注：勾选此项时，必须记录样本号，附加简短说明。

## **V1版6大分类**

### **1. 领域认知多义性 (Domain-Cognitive Polysemy)**

**定义：** 开发者使用了专业背景、局部化术语、黑话或隐喻。与 LLM 预训练语料中的通用语义或者编程方向语义分布不一致，从而引发代码方向错误。

**映射 NLP ：**Lexical (词汇)、Idiomatic (习语/隐喻)

**产生原因**：开发者在特定的学术研究或工程项目中，会形成高度局部化的领域知识，作为认知捷径。

**大白话：**用户在 Prompt 里用了“黑话”、“简称”或“行话”。

• **正例 1：**财务背景开发者输入：“计算这笔交易的Buffer。”（在计算机领域Buffer是“缓冲区”。）
• **正例 2：**用户让模型“写一个 Node”或“Kill the child process”。（模型可能混淆 DOM 节点与后端环境，或对操作系统级隐喻（孤儿/子进程）产生字面误解。）
• **易混淆负例：** “写一个单元测试用例。” (此为通用编程概念，不属于局部多义词，若信息不够应归类于习惯性语境缺失。）

### **2. 逻辑结构映射误差 (Structural Logic Misalignment)**

**定义：** 开发者使用扁平、启发式的自然语言来描述复杂的控制流（条件、循环）或算子优先级。由于句法依存模糊或量词作用域不清，导致布尔逻辑或执行拓扑允许多种解释。

**映射 NLP ：**Syntactic (句法)、Scopal (范围)、Collective/Distributive (集合/分配)

**产生原因：**人类倾向于使用扁平、模糊的自然语言来描述包含多重嵌套的控制流，而不习惯像编译器那样严格定义作用域。

**大白话：**开发者不够严谨，用自然语言描述逻辑关系。

• **正例 1：** “如果 user 为空并且 token 过期或者请求来源不是白名单，就拦截请求。” (算子优先级极度模糊：是 (A && B) || C 还是 A && (B || C)？）
• **正例 2：** “检查一下这些上传的文件大小是否超过 10MB。” (集合分配错误，是计算所有文件的总和与10MB比较，还是分别比较？)
• **易混淆负例：** “用 for 循环遍历列表，如果 item 为空就还按我的习惯处理。” (无逻辑歧义，但习惯性语境缺失。)

### **3. 习惯性语境缺失 (Habitual Context Omission)**

**定义：** 开发者无意识地省略了当前项目特有的数据结构、前置依赖或隐性共识，导致模型面临“缺少关键拼图”的局面，无法写出可直接运行的代码。

**映射 NLP ：**Elliptical (省略)

**产生原因：**知识的诅咒 ，当开发者长期沉浸在一个架构中时，大量的工作记忆会被内化为隐性共识，从而在沟通中无意识地省略这些信息。

**大白话：**开发者脑子里有图纸，但嘴上没说，默认大模型知道。

• **正例 1：** “写一个快排函数”。（是原地排序还是返回新数组？升序还是降序？处理何种数据类型？）
• **正例 2：** “按照老规矩，把这部分业务逻辑抽离成一个 Service。” (“老规矩”是团队内部隐性共识，LLM 无法得知具体的架构规范。)
• **易混淆负例：** “用 SQLAlchemy 写一个连接 PostgreSQL 的单例模式。” (上下文自闭环，未缺失关键数据结构。若没写连接池配置，属于“隐式约束不足”)。

### **4. 系统边界认知错位 (System Boundary Misconception)**

**定义：** 用户预设了 LLM/Agent 具备越权的环境感知能力（如读取本地文件）或越权操作能力（如直接修改系统配置）。

**映射 NLP ：**Presuppositional (预设)

**产生原因：**人类在与表现出高智能的 Agent 交互时，容易产生拟人化倾向，错估系统的感知边界和操作权限。

**大白话：**开发者把大模型当成了“全能神”。

• **正例 1：** “帮我看看桌面上那个 error.log 为什么一直报错。” (错误预设模型能读取本地文件系统。)
• **正例 2：** “直接把服务器上的 Nginx 重启一下吧。” (错误预设模型当前具备 SSH 执行权限，且未提供任何鉴权上下文。)
• **易混淆负例：** “写一个自动重启 Nginx 的 bash 脚本。” (指令只是要求“生成文本”，并没有要求模型“去执行”，因此无歧义)。

### **5. 交互上下文失焦 (Conversational Memory Disalignment)**

**定义：** 发生于多轮对话或长文本中。用户使用了代词（它、那个）或混淆了抽象类与具体实例，导致llm目标丢失。

**映射 NLP ：**Coreferential (指代)、Type/Token (类型/实例)

**产生原因：**人类的注意力焦点与模型的上下文窗口的不对齐。用户认为交互是一个连续的、有情感偏好的过程，而模型通常基于最近优先原则。

**大白话：**多轮对话中代词滥用。

• **正例 1：** “把它改成异步非阻塞的写法。” (“它”指代不明，可能是上一轮对话的函数，也可能是上上轮的某个模块。)
• **正例 2：** “像刚才处理 User 模型那样，处理一下当前的逻辑。” (高度依赖历史对话窗口，若历史存在多个“处理”动作，将产生歧义。)
• **易混淆负例：**无

### **6. 隐式约束说明不足 (Implicit Constraint Under-specification)**

**定义：** 开发者处于“目标导向”模式，只描述了理想情况，却将非功能性需求（NFR，如安全性、异常处理、边界条件、鲁棒性）视为理所当然的隐式约定。

**映射 NLP ：**Implicative (蕴含)、Generic/Non-Generic (泛指/特指)

**产生原因：**人类在描述需求时，倾向于描述“我要实现什么”，而将安全性、鲁棒性以及泛化边界视为理所当然的默认标准。

**大白话：**开发者认为llm写代码的标准应该天然与自己对齐，包含代码复杂度和边界情况是否需要考虑。

• **正例 1：** “写一个接收用户密码并存入数据库的 API。” (缺失了密码必须Hash 加盐、防 SQL 注入等安全约束，导致生成的明文存储代码在工程中不可用。)
• **正例 2：** “写个爬虫把这个新闻网站首页的标题都抓下来。” (隐式缺失了请求头伪装、并发频率限制、防反爬策略等工程化约束。)
• **易混淆负例：** “在这个请求里加上鉴权 Token。” (知道要加 Token，但没说明 Token 怎么获取、放 Header 还是 Body，这属于“习惯性语境缺失”，因为缺少具体的数据交互结构，而不是缺少安全意识。)

### **易混淆的概念：**

1.隐式约束说明不足中的Generic/Non-generic (泛指/特指)：是关于“规律泛化”与“具体案例”的冲突，
用户这句话，是想让我写一个通用规则，还是仅仅处理当前这一个特例？

逻辑结构映射误差中的Type/Token (类型/实例)：是关于Class与Instance/Object的冲突，比如：

- Prompt: “复制刚才那个 User 节点，并把它加到树的最右边。”
- Type (类型) 理解： LLM 认为你想要的是一个“同款”，于是 new User(old_node.data)，在内存里开辟了一块新空间（Deep Copy），放进树里。
- Token (实例) 理解： LLM 认为你说的是绝对的“同一个实体”，于是直接把刚才那个节点的指针/引用（Reference）塞到了树的最右边。这可能会导致后续操作产生严重的副作用（修改一个，两个都变）。

2.逻辑结构映射误差中Collective/Distributive(集合/分配歧义)怎么理解？

- Prompt: “把这些选中的商品发送给计算运费的 API。”
- Collective (集合 - Batching)： 大模型写了一段代码，把整个 List 打包成一个 JSON Array [item1, item2, item3]，发了 1 次网络请求。（这通常是正确的且性能好）。
- Distributive (分配 - Iteration)： 大模型写了一个 for 循环，遍历这个 List，发了 N 次网络请求。（这可能导致 API 限流或系统崩溃）。

针对包含“列表、数组、群组、复数名词”的 Prompt 时，大模型到底应该把它们‘当成一个整体打包处理（Batch/List 级别操作）’，还是应该‘写个循环挨个处理（Element 级别操作）’？如果这两种写法都能解释得通，那就是 Collective/Distributive 歧义。

3.隐式约束说明不足和习惯性语境缺失应该怎么区分？

关键：缺的到底是“基本积木”还是“安全护栏（非核心功能规则）”？

习惯性语境缺失 —— 缺“积木”：开发者省略了物理/逻辑上的必需品（比如具体的 JSON 结构、数据库表名、团队内部封装的组件名）。代码连最基本的“Happy Path”都跑不通。

隐式约束说明不足—— 缺“护栏”：开发者省略了非功能性需求 (NFR)（比如防注入、错误重试、分页、密码加密）。这往往是一段脆弱的代码，一旦放到真实的生产环境，立刻就会因为没有防御机制而崩溃或被黑客攻击。

4.领域认知多义性 vs 习惯性语境缺失—— “黑话”和“缺少图纸”的微妙边界
混淆场景： Prompt 里出现了一个大写字母开头的专有名词，比如 “把这个传给 MagicBox 处理。” 这到底是业务上的“黑话”（多义性），还是没交代类的定义（缺语境）？
区分原则：
    ◦ 多义性（自信地写错）： 这个词在通用英语/编程里有别的意思。大模型以为自己懂了，非常自信地写了一段代码，但完全偏离了开发者的真实意图。比如 “把这个孤儿杀掉 (Kill the orphan)”（大模型可能去写操作系统的孤儿进程清理，但开发者说的可能是游戏引擎里没有父节点的 UI 元素）。
    ◦ 缺语境（懵逼地瞎编）： 这个词（比如 MagicBox、TbUser_Auth_V2）完全是团队自己造的。大模型明知道自己不知道这是啥结构，只能硬着头皮瞎编里面的方法。
5. 歧义 (Ambiguous) vs 困难 (Complex/Hard)—— Not Ambiguous 标签的纯洁性
看到大模型生成了一堆狗屎代码时，会下意识地认为“Prompt 肯定有问题”，从而强行给 Prompt 挑刺打标签。Prompt 的要求极其精确、逻辑严密，在计算机科学中只有唯一的正确实现方向。如果大模型写错了，那是它推理能力不行，不能算作 Prompt 的歧义！

## **标签标注决策树**

标注者必须按照以下“过滤管道”的顺序对每一条 Prompt 进行判断。遇到符合条件的节点，**勾选对应标签，并继续向下判断，直到过程结束。**
**执行 SOP：**
1. 【前置拦截】 这段指令是否只有唯一正确解？模型写错纯粹是因为它太笨？
    Yes 则勾选[Not Ambiguous]，结束标注。
    No 则进入标签检测 
2. 【词汇检测】 是否使用了容易被模型误解的黑话、隐喻或特定领域专有术语？
    Yes 则 勾选 [领域认知多义性] 否则 继续下一步
3. 【逻辑检测】 是否存在“和/或/非”算子嵌套模糊、条件执行顺序不清、量词作用域不明的情况？
    Yes 则 勾选 [逻辑结构映射误差] 否则继续下一步
4. 【指代检测】 是否使用了“它”、“刚才那个”等代词，导致依赖的历史对话上下文不明？
    Yes 则 勾选 [交互上下文失焦] 否则 继续下一步
5. 【省略检测】 是否理所当然地省略了特定的 JSON 结构、数据库表结构或前置参数？
   Yes 则 勾选 [习惯性语境缺失] 否则 继续下一步
6. 【权限检测】 是否误以为模型能联网、能看本地文件、能直接操作你的电脑？
   Yes则 勾选 [系统边界认知错位] 否则 继续下一步
7.【工程检测】业务逻辑看起来很完美，但是否毫无防备？（缺异常处理、缺安全机制、缺性能考虑）？
    Yes 则勾选 [隐式约束说明不足] 否则 继续下一步
8. 【兜底检测】 走完上述流程，你觉得仍有歧义未被覆盖？
    Yes 则 勾选 [Other]，并在输入框写明理由 ，完成后 结束标注。
